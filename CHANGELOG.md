# История изменений

### 1.1.59 -- 2026-06-19
- **Сборка soft-float `awg` починена (релиз 1.1.58 не собрался).** В 1.1.58 шаг сборки soft-float `awg` падал в CI (скачивание musl-тулчейна + запись в `/opt` под обычным пользователем раннера) — поэтому у релиза 1.1.58 не было ни одного `.ipk`. Теперь `awg` для `armv7-2.6` собирается через образ **dockcross `linux-armv5-musl`** (gcc + musl + ARMv5 без VFP + soft-float ABI, статически) — надёжно тянется с Docker Hub и не зависит от записи в `/opt`. Функционально идентично 1.1.58: тот же soft-float `awg` для процессоров без аппаратного FPU (RT-AC68U) + CI-проверка против hard-float

### 1.1.58 -- 2026-06-19
- **Настоящее исправление `Illegal instruction` на RT-AC68U (и других роутерах без аппаратного FPU).** Диагностика (`diag`) показала главное: Cortex-A9 в RT-AC68U **не имеет VFP** — в `/proc/cpuinfo` Features только `swp half thumb fastmult edsp`, без `vfp`/`neon`. Значит **любой** бинарник с hard-float (VFP) падает с `Illegal instruction` на первой же float-инструкции (поэтому `awg` падал даже на выводе справки, не только в `genkey`). В 1.1.55 я понизил уровень ISA (ARMv7→ARMv6), но сборка Alpine `armhf` всё равно **hard-float** — отсюда «без изменений». Теперь `awg` для пакета `armv7-2.6` собирается **soft-float** статически через musl-кросс-тулчейн `arm-linux-musleabi` — ровно как Go-демон с `GOARM=5`, который изначально работал. В CI добавлена проверка: сборка падает, если `awg` вдруг снова окажется hard-float. Пакеты `aarch64-3.10` и `armv7-3.2` не изменились

### 1.1.57 -- 2026-06-19
- **Диагностика запуска: команда `diag` + подробный лог старта.** Чтобы быстро понять, почему VPN не стартует на конкретном роутере, добавлена команда `amneziawg.sh diag` (и `S99amneziawg diag`). Она печатает: платформу (`uname`, CPU из `/proc/cpuinfo`, `opkg print-architecture`), установленный пакет (`opkg list-installed`), архитектуру бинарников `amneziawg-go` и `awg` — определяется чтением ELF-заголовка через busybox `dd`/`od`, без `file`/`readelf` (класс ELF32/64, машина ARM/AARCH64, EABI/float-ABI), и живые пробы: запускает `amneziawg-go --version`, `awg` и `awg genkey`, показывая код возврата и явно отмечая `SIGILL (Illegal instruction)`. Кроме того, сам старт теперь пишет в лог строку с платформой и архитектурой обоих бинарников, а сбой `awg setconf` отличает SIGILL (код 132 = бинарник под неподходящую архитектуру) от прочих ошибок

### 1.1.56 -- 2026-06-19
- **Значок «AWG» в шапке корректно встаёт на классической теме** (RT-AC68U, RT-AX88U и др.). Раньше значок искал только ROG-контейнер `#status_block` (есть лишь на GT-AX11000 и подобных), а на классической теме, где строка статуса — таблица из ячеек `<td>`, не находил его и «улетал» из шапки (отображался у бокового меню). Теперь размещение зависит от темы: ROG → первым элементом внутри `#status_block`; классическая → отдельной ячейкой `<td>` первой иконкой в строке статуса (строка ищется по любой присутствующей иконке — notification/QoS/wifi/connect/usb); если разметка незнакомая — плавающая плашка справа сверху

### 1.1.55 -- 2026-06-19
- **Исправлен запуск на старых роутерах (RT-AC68U, RT-AC66U и др. на Cortex-A9 / `armv7-2.6`).** Утилита `awg` падала с `Illegal instruction` на старте (шаг `awg setconf`), хотя `amneziawg-go` запускался и интерфейс поднимался. Причина: для пакета `armv7-2.6` Go-демон собирался консервативно (`GOARM=5`), а C-утилита `awg` бралась общей с пакетом `armv7-3.2` — сборка Alpine `linux/arm/v7` (ARMv7-A, с NEON и аппаратным делением `udiv`/`sdiv`), которых у Cortex-A9 нет. Теперь для `armv7-2.6` собирается отдельный `awg` под Alpine `linux/arm/v6` (ARMv6 + VFPv2 — строгое подмножество инструкций, идёт и на A9, и на всех новых ARMv7-ядрах). Пакеты `aarch64-3.10` и `armv7-3.2` не изменились

### 1.1.54 -- 2026-06-19
- **Загрузка обновлений через VPN (обход блокировок сразу).** В настройках появился блок **«Загрузка через VPN (обход блокировок)»** с двумя галочками: **«Geo-списки через VPN»** и **«Обновление программы через VPN»**. Когда галочка включена и туннель поднят, загрузка идёт через активный AWG-интерфейс — запросы к GitHub / jsDelivr выходят через VPN-сервер в обход региональных блокировок (можно ставить обновления прямо с GitHub и сверять SHA256 по GitHub API). Реализовано привязкой `curl --interface <IP awg0>`: трафик от адреса туннеля уже маршрутизируется через VPN существующим правилом `ip rule from <awg0> lookup 300`. Если VPN выключен — загрузка идёт напрямую, как раньше (никакого тупика при самообновлении). DNS-резолвинг остаётся системным, поэтому обход работает для блокировок по IP/TCP, но не от DNS-подмены. Галочки действуют сразу при нажатии **Update Now** / **Установить**, даже без отдельного Apply

### 1.1.53 -- 2026-06-19
- **Индикатор в шапке ставится надёжнее.** Значок «● AWG» теперь вставляется первым элементом в строку статуса (`#status_block`), без привязки к иконке QoS (`#bwdpi_status`). Прежняя привязка к QoS ломалась, если QoS (или другая иконка-якорь) скрыт или отсутствует в зависимости от конфигурации роутера; левый слот строки статуса — единственный надёжный

### 1.1.52 -- 2026-06-19
- **Выбор способа установки в окне обновления.** Вместо поля «версия» теперь выпадающий список **«Установить»** с тремя режимами: **Автоматически (последняя)** — ставит последнюю опубликованную версию; **Выбрать версию** — появляется поле для ввода версии `X.Y.Z`; **Вручную через файл** — появляется выбор файла `.ipk` с локального диска
- **Ручная установка `.ipk` из браузера.** Можно загрузить собственный/локальный пакет `.ipk` прямо через веб-интерфейс, без SSH. Прошивочный httpd ограничивает обычный POST до ~64 КБ и читает его построчно (бинарь не проходит), поэтому файл кодируется в base64 в браузере и передаётся на роутер частями (с подтверждением и токеном на каждую часть, повтор при таймауте); бэкенд собирает их, проверяет **точную длину + CRC gzip + структуру `.ipk` (`control.tar.gz`)** и только потом ставит через `opkg`. Повреждённая или оборванная загрузка не пройдёт проверку и не установится. Прогресс показывается полосой загрузки
- **Версия пакета (`PKG_VERSION`) синхронизирована с `AWG_VERSION`** (1.1.52), чтобы имя ассета `.ipk` совпадало с URL, который собирает авто-обновление

### 1.1.51 -- 2026-06-19
- **Geo Antifilter — готовые списки РКН.** Под блоком GeoSite появился блок **«Geo Antifilter»** с галочками для списков заблокированных в РФ ресурсов с [antifilter.download](https://antifilter.download/). IP-списки (`allyouneed`, `community`, `ipsum`, `subnet`, `ip`, `ipresolve`) грузятся в тот же ipset `awg_dst`, что и GeoIP, и маршрутизируются через VPN; небольшой список `community` доменов уходит в dnsmasq. Рекомендуемый выбор — `allyouneed` (~15K подсетей, = `ipsum` + `subnet`). Обновляются той же кнопкой **Update Now** и автообновлением, что и остальные гео-списки
- **Лимит ipset поднят** со 131072 до 262144, чтобы крупные списки antifilter (один `ipresolve` ~154K) помещались вместе с GeoIP/GeoSite. Полный `domains.lst` (1.4M доменов / 27 МБ) намеренно не подключается — он слишком велик для dnsmasq на роутере

Меньше кнопок, меньше путаницы. Если уже последняя — кнопка сообщает об этом, а не запускает пустое обновление

### 1.1.49 -- 2026-06-19
- **Исправлено ложное «доступно обновление» на более старую версию.** Раньше обновление считалось доступным при любом несовпадении версий, поэтому при устаревшем ответе jsDelivr (например, 1.1.46, пока кэш data API не догнал реальную версию) на свежей сборке показывалось «Доступно обновление: v1.1.46», а кнопка «Обновить» могла откатить назад. Теперь версии сравниваются по номеру — обновление предлагается, только если оно строго новее установленного. Плюс запрос версии к jsDelivr идёт с cache-buster, чтобы браузер не показывал устаревший кэш

### 1.1.48 -- 2026-06-19
- **Проверка SHA256 вернулась как «по возможности».** Контрольная сумма берётся из GitHub API (`digest` ассета релиза): если API доступен — пакет проверяется (несовпадение = отказ от установки), если заблокирован — проверка пропускается, а установка продолжается. Без ветки `checksums` и без отказа при загрузке через зеркало — только опциональная сверка по API. В лог пишется, проверена сумма или пропущена

### 1.1.47 -- 2026-06-19
- **`update` не «зависает» в регионах с заблокированным GitHub.** Раньше прямая загрузка с GitHub висела до ~9 минут (`--max-time 180 --retry 2`) перед переходом на зеркало — выглядело как зависание. Теперь застрявшее соединение обрывается за ~15 с (`--speed-time`) и сразу уходит на рабочее зеркало
- **Подробный лог обновления.** Каждый шаг пишется в лог: определение версии, целевая версия, попытка по каждому источнику (`github.com` / зеркало) с результатом, размер скачанного, остановка VPN, установка opkg, завершение — видно, на каком шаге что-то застряло
- **Видно, когда обновлять нечего.** Если установленная версия совпадает с последней, в лог пишется «already on vX — nothing to install» (раньше при этом была тишина, и казалось, что команда зависла)
- **Проверка SHA256 убрана полностью** — из скриптов, из workflow и из репозитория (ветка `checksums` удалена). Обновление упрощено: пакет берётся по HTTPS (напрямую или через зеркало) без сверки контрольной суммы
- **Установка конкретной версии.** В окне обновления появилось поле «Установить конкретную версию» — можно поставить любой опубликованный релиз. Из консоли: `S99amneziawg update 1.1.45` (или `amneziawg.sh update 1.1.45`). Версия передаётся явно, поэтому лаг резолва jsDelivr на неё не влияет
- **Новый релиз сразу виден роутеру.** После публикации workflow регистрирует тег в jsDelivr, поэтому проверка обновлений (где версия берётся из jsDelivr при заблокированном `api.github.com`) видит новую версию без задержки на кэш-краул

### 1.1.44 -- 2026-06-19
- **Установка и обновление работают без `api.github.com`.** В ряде регионов GitHub API заблокирован, из-за чего онлайн-установщик и проверка обновлений падали с «Cannot reach GitHub API». Теперь версия релиза определяется по цепочке: GitHub API → jsDelivr data API → `cdn.jsdelivr.net` (тот же хост, откуда тянется сам установщик), а имя `.ipk` собирается детерминированно — установщик, проверка обновлений в веб-интерфейсе и автообновление в роутере больше не зависят от доступности `api.github.com`
- **Безопасное обновление через прокси-зеркало.** При релизе SHA256 пакетов публикуются в отдельную ветку `checksums`; роутер берёт нужную сумму через jsDelivr (git-контент — зеркало `ghproxy`/`gh-proxy` подменить её не может) и проверяет скачанный пакет. Раньше обновление через зеркало без опубликованной суммы отклонялось; теперь оно безопасно работает и там, где недоступны и API, и хост релизов GitHub
