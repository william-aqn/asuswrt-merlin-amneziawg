# История изменений

### 1.1.98 -- 2026-06-21
- **«Режим совместимости» с DPI-обходами: распознавание b4 и nftables + включение по умолчанию для новых установок.** Автоопределение соседних DPI/прокси-утилит дополнено **b4** (по имени процесса) и следом NFQUEUE/TPROXY в **nftables** -- раньше учитывались только правила iptables, поэтому на nft-прошивках соседний инструмент мог быть пропущен, и перехват :53 ломал DNS. Бывшая галочка «Не перехватывать DNS» переименована в **«Режим совместимости»** и при чистой установке включается по умолчанию (у уже установленных поведение не меняется до ручного включения), чтобы даже неизвестная утилита не могла оставить сеть без интернета. В «Скопировать диагностику» добавлен блок сосуществования (обнаруженная утилита, backend NFQUEUE, conntrack, пересечение fwmark 0x100), а в журнал -- предупреждение о риске нехватки памяти (OOM) при одновременной работе DPI-утилиты и гео-списков на роутерах с малым ОЗУ.

### 1.1.97 -- 2026-06-21
- **В журнале теперь видна дата, а не только время.** Строки журнала (и в окне «Журнал», и в «Скопировать диагностику») получают полную метку времени `ГГГГ-ММ-ДД ЧЧ:ММ:СС` вместо одного только времени — удобно при разборе логов, охватывающих несколько дней или переход через полночь (фоновые загрузки гео-списков, watchdog, авто-откат DEADMAN).

### 1.1.96 -- 2026-06-21
- **Гео-поля стали растягиваемыми (textarea).** Поля «Списки сервисов GeoIP», «Списки сервисов GeoSite», «Свои домены» и «Свои IP / подсети» теперь многострочные и тянутся за угол по высоте. Можно вводить значения и через запятую, и с новой строки; автодополнение у списков сервисов сохранено. При сохранении значение нормализуется в список через запятую (пробелы/переводы строк убираются).
- **Новый раздел «GeoCustom — собственные файлы».** Под блоком GeoSite добавлены два механизма для своих списков маршрутизации: (1) **свои файлы** — именованные списки, которые можно вставить/отредактировать прямо в интерфейсе или загрузить из локального файла (имя автоматически приводится к `[a-zA-Z0-9]`, остальное → `_`); (2) **URL-источники** — повторяемые поля «+» со ссылками на скачиваемые списки. Содержимое каждого списка определяется построчно: домен (`example.com`) → маршрут через DNS, IP или подсеть CIDR (`1.2.3.0/24`) → в ipset; строки с `#` — комментарии. Формат описан в подсказке. Свои файлы хранятся в настройках и пересобираются при «Применить»; URL-источники скачиваются при «Обновить списки»/«Применить» (с тем же резервированием зеркал и прокси, что и встроенные списки) и удаляются при удалении ссылки.
- **Анализатор трафика: добавление в свои домены/IP.** В окне анализа у каждой строки появился чекбокс (и «выбрать все»); кнопка «+ В свои домены/IP» добавляет отмеченные строки в поля «Свои домены»/«Свои IP» одной кнопкой — для каждой строки берётся домен (если он есть), иначе IP, без дублей. Выбор сохраняется при автообновлении списка (раз в 1,5 с). После добавления не забудьте «Применить».

### 1.1.92 -- 2026-06-21
- **Safari «Сохранить пароль» — окончательно.** Настоящая причина была не в названиях полей, а в самом `-webkit-text-security: disc`: WebKit считает любое поле с `text-security` полем пароля. Маскировка теперь делается только встроенным шрифтом-дисками (визуально без изменений, работает во всех браузерах), без `-webkit-text-security`. Дополнительно убрали `scope="row"` у заголовков полей ключей, чтобы у замаскированных полей не осталось вообще никакой ассоциации «поле ↔ подпись».

### 1.1.91 -- 2026-06-21
- **Анализ трафика: видны DNS-запросы устройства (намерение «до резолва»).** Раньше окно показывало только реальные соединения (из conntrack), названные по домену, — то есть уже *после* резолва, по факту подключения. Теперь отдельными строками с меткой **DNS** выводятся домены, которые устройство пытается разрешить, ещё до установки соединения, с вердиктом **напрямую / через VPN (Geo)**. Вердикт берётся по разрешённым IPv4 (для `vpn_geo` — проверка адреса в гео-наборе `ipset`), CNAME-цепочки корректно сопоставляются по id DNS-запроса, домены берутся из запросов любого типа (A/AAAA/HTTPS). Домены видны только для устройств, использующих роутер как DNS. Проверено на роутере на живом трафике.

### 1.1.90 -- 2026-06-21
- **Safari больше не предлагает «Сохранить пароль» (продолжение 1.1.89).** Нейтрализации `id`/`class` и ключей настроек оказалось мало: Safari всё ещё реагировал на `aria-label="Private Key (секрет)"` у замаскированных полей (он переустанавливался при каждой загрузке и смене языка через `data-i18n-aria`). Убрали `aria-label` и `data-i18n-aria` у полей приватного / публичного / preshared-ключей и удалили лишние словарные строки; видимые заголовки строк получили `scope="row"`, чтобы скринридеры по-прежнему озвучивали назначение поля. Класс маскировки переименован `awg-masked` → `awg-dotted`.
- **Мелкие улучшения интерфейса.** В окне анализа трафика теперь видны имя устройства и его `IP·MAC` (берутся из списка клиентов роутера); окно выбора устройства закрывается по `Escape`; кнопка удаления устройства — иконкой (как у кнопки анализа).

### 1.1.89 -- 2026-06-21
- **Safari больше не предлагает «Сохранить пароль» на странице настроек.** Поля приватного ключа / публичного ключа / preshared-ключа задавались через `id`/`class` со словами `key`/`secret`/`private`, на которые менеджер паролей Safari реагирует как на форму входа. Теперь имена полей нейтральны по всей цепочке: `id` в разметке = ключ в `custom_settings` = ключ `get_setting` на бэкенде (`awg_iface_p1` / `awg_peer_p1` / `awg_peer_p2`), а класс маскировки переименован `awg-secret` → `awg-masked`. Существующие конфигурации переносятся автоматически при обновлении: значения из старых ключей подхватываются на фронте, а бэкенд одноразово переименовывает их в файле настроек и удаляет дубликат секрета под старым именем.

### 1.1.88 -- 2026-06-21
- **Исправлен анализ трафика — лог был пустым.** Захват соединений полагался на утилиту `conntrack`, которой нет в стоковой прошивке Asuswrt-Merlin, поэтому ничего не попадало в лог. Теперь соединения читаются напрямую из таблицы ядра `/proc/net/nf_conntrack` (с запасным `/proc/net/ip_conntrack`), а протокол берётся из поля L4. Проверено на роутере: соединения устройства появляются сразу с вердиктом **напрямую / через VPN (Geo)**; доменные имена подставляются для адресов, разрешённых во время захвата.

### 1.1.87 -- 2026-06-21
- **Анализ трафика устройства.** В таблице «Правила устройств» добавлена колонка «Анализ» с кнопкой: открывает окно со Старт/Стоп и живым логом запросов выбранного устройства — у каждого запроса показано, идёт он **напрямую** или **через VPN/Geo**. Маршрут определяется по применённой политике устройства и проверке адреса назначения в гео-наборе (`ipset test`) — то есть так же, как решает ядро. Доменные имена берутся из временного лога запросов dnsmasq (на время захвата кратко перезапускается DNS; имена видны только для устройств, использующих роутер как DNS), а сами соединения — из `conntrack` (ловятся в т.ч. приложения с «зашитыми» IP). Захват сам останавливается через 10 минут и полностью убирается при остановке VPN / удалении (логирование DNS не остаётся включённым).

### 1.1.83 -- 2026-06-20
- **«Последнее рукопожатие» теперь живое.** Бэкенд отдаёт в статус-JSON сырое время рукопожатия (`hs_epoch`) и счётчики трафика (`rx_bytes`/`tx_bytes`) рядом с готовыми строками; страница считает «N с/мин/ч назад» в браузере и обновляет счётчик **каждую секунду**, не дожидаясь опроса. Раньше значение запекалось строкой при старте и «зависало» (например, «8 с назад» навсегда, даже после F5).
- **Трафик и рукопожатие обновляются в фоне.** Добавлен лёгкий крон `awg_status` (раз в минуту), который перегенерирует статус между действиями пользователя, — поэтому счётчики приёма/передачи больше не стоят на месте. Крон только читает (`awg show` + запись файла), не берёт блокировку и не дёргает `notify_rc`; живёт и снимается строго вместе с watchdog. То же исправление работает и для мини-панели виджета в шапке.
- Внутреннее: запись статуса в PID-уникальный временный файл (`$$`) + подчистка «висяков», чтобы фоновый крон и действие пользователя не мешали друг другу.
- **ipset без утечек.** Набор, созданный самим аддоном, теперь корректно удаляется при остановке **и при смене имени** (старый «висяк» зачищается). Набор, уже созданный другим подключением/инструментом, аддон не уничтожает (общий) — определяется по тому, кто реально создал набор.
- **Виджет.** Статусы и кнопки мини-панели приведены к терминам основной страницы: «Подключено/Остановлено», «Запустить/Остановить», переходное состояние — «Подключение…/Остановка…».

Теперь проверка `tunnel_alive` делает несколько быстрых проб по двум anycast-целям (8.8.8.8 и 1.1.1.1) и считает туннель живым, если ответил **хоть один** пакет; рестарт — только при устойчивой потере всех проб. На здоровом туннеле выходит на первом же ответе (быстро)
- **Возвращён `filter-AAAA` при выключенном IPv6.** В 1.1.69 запись `filter-AAAA` была ошибочно завязана на `ipv6_service != disabled`. У кого IPv6 выключен (частый случай), `filter-AAAA` переставал писаться → клиенты с dual-stack (телевизоры, браузеры — Happy Eyeballs) сначала пробовали мёртвые AAAA-адреса, висли на таймаутах и только потом откатывались на IPv4 → тормоза/буферизация. Теперь `filter-AAAA` пишется всегда, когда галка включена И активна VPN/geo-политика, независимо от состояния IPv6 (он полезен и при выключенном IPv6 — заставляет идти по IPv4, который и маршрутизируется по geo). Гейт «не резать AAAA всему LAN, если ничего не в VPN» сохранён
- **Откат маскирования fwmark (`0x100/0x100` → `0x100`).** В 1.1.69 метка ставилась маской, чтобы не затирать биты соседних инструментов — но это опциональное упрочнение трогало «горячий» путь помеченного трафика, а поддержка синтаксиса маски на старых ядрах/iproute2 не гарантирована. ### 1.1.69 -- 2026-06-19
По итогам полного аудита сетевого взаимодействия (iptables / маршрутизация / DNS / сосуществование / гонки жизненного цикла). Сводка подтверждённых проблем и их устранение:
- **Kill-switch (новая галочка в UI, по умолчанию ВЫКЛ).** Раньше при аварийной смерти туннеля (краш `amneziawg-go` / OOM / ручное `ip link del`, минуя `do_stop`) ядро вычищало маршруты таблицы 300 (`dev awg0`), но ip-правила `fwmark`/`from` оставались — трафик устройств с политикой «VPN» **проваливался в WAN открытым текстом** (fail-open). Добавлен device-независимый маршрут `blackhole default table 300 metric 1000`, который выживает при исчезновении `awg0`: пока туннель жив, выигрывают более точные `/1`-маршруты; когда `awg0` пропал — трафик дропается (fail-closed), а не утекает. Включается галочкой **«Kill-switch»** (блок настроек рядом с «Prevent IPv6 Leaks»); по умолчанию выключено, чтобы не менять прежнее поведение без явного согласия. Состояние отдаётся в статус-JSON полем `killswitch`
- **Авто-откат больше не убивает самовосстановление.** `cleanup_firewall` удаляла cron `awg_watchdog`, а ставил его обратно только `setup_firewall`. Откат health-check и дедмана вызывают `do_stop` без последующего `do_start` — в результате после авто-отката туннель лежал **и** watchdog-cron был удалён, поэтому 5-минутный авто-ретрай (ради которого делались 1.1.66–1.1.68) пропадал — «VPN сам не переподключается». Теперь удаление watchdog-cron вынесено из `cleanup_firewall` и происходит **только при намеренной остановке пользователем** (`awgstop`/CLI `stop`/uninstall). Авто-откаты (дедман зовёт новый внутренний `stop_auto`) cron сохраняют
- **«Применить» (`awgsaveconf`) и фоновая до-загрузка geo теперь под локом.** Это были единственные пути, перестраивавшие firewall **без** блокировки — гонка с `do_firewall_restart` (хук `firewall-start`) могла дать `ipset destroy: set is in use`, полусобранную mangle-цепочку или осиротевший DNAT `:53`. Теперь `setup_firewall` на этих путях выполняется под общим локом (`acquire_lock`), а при «Применить» дополнительно взводится LAN-дедман — как при полном старте
- **Утечка IPv6 после «Применить» устранена.** `setup_ipv6_block` вызывался только из `do_start`/`do_firewall_restart`, а его снос (`cleanup_ipv6_block`) — внутри `cleanup_firewall`. Поэтому «Применить» на работающем туннеле сносило REJECT-правила IPv6 и не ставило обратно (особенно опасно в режиме сосуществования, где `filter-AAAA` обходится). Теперь `setup_ipv6_block` идемпотентен (`-C`-гард) и вызывается в конце `setup_firewall` — единой точки пересборки, которую проходят все три пути
- **Сосуществование DNS теперь самосогласуется (watchdog).** Решение «перехватывать ли `:53`» принималось один раз в `setup_firewall` и устаревало: если DPI/прокси-тул (Xray/zapret) стартовал **после** AWG (типично на загрузке), перехват оставался и конфликтовал — «интернет умер после ребута», а `dns_ok` этого не видит (опрашивает `127.0.0.1`, минуя DNAT на br0). Watchdog теперь на каждом тике сверяет состояние: снимает наш перехват `:53`, если рядом появился DPI-тул/включён `awg_no_dns_intercept`/активны DNS-фичи прошивки; и наоборот ставит перехват, когда DPI-тул убрали или `dnsmasq` поднялся после загрузки. Добавлен детект **DNSFilter / DNS Director / DoT прошивки** (`fw_dns_redirect_active`) — наш глобальный DNAT `:53` больше не перекрывает их молча
- **Предупреждение при `vpn_all` + соседний DPI-тул.** Если рядом обнаружен прокси/DPI-тул и политика по умолчанию `vpn_all` (или есть клиент `vpn_all`) — пишется явное предупреждение в лог и поле `coexist_warn` в статус-JSON. (Уточнение: собственный egress прокси-демона **не** крадётся — наша mangle-цепочка только в PREROUTING; речь о LAN-трафике, который `vpn_all` уводит в туннель)
- **Geo/ipset:** правила dnsmasq `ipset=/.../awg_dst` больше не пишутся, если набор не создан (старое ядро без `ip_set_hash_net` / конфликт libipset) — раньше это давало ошибку на каждый DNS-ответ и молчаливую смерть geo-по-доменам; предупреждение «ipset full» теперь сравнивается с **эффективным** лимитом (на низкоОЗУ-роутерах оно вообще не срабатывало); `download_geosite` валидирует YAML (наличие категорий) перед заменой — флапающее зеркало с HTTP-200 и HTML-ошибкой больше не затирает хорошую базу; `filter-AAAA` пишется только когда IPv6 включён И задействована VPN/geo-политика (не режет AAAA всему LAN, включая `direct`-устройства)

### 1.1.64 -- 2026-06-19
- **Geo-маршрутизация починена по-настоящему: конфликт `libipset` (Entware vs прошивка).** Логирование из 1.1.61 наконец показало настоящую причину `ipset awg_dst creation failed`: `ipset: /usr/lib/libipset.so.13: version LIBIPSET_4.11 not found (required by ipset)`. Аддон стартует из httpd/service-event без Entware-профиля (`/opt/etc/profile`), поэтому Entware-бинарь `/opt/sbin/ipset` (ему нужен символ `LIBIPSET_4.11`) подхватывает старую библиотеку прошивки `/usr/lib/libipset.so.13`, где этого символа нет, и **падает на этапе динамической линковки** — а значит падали ВСЕ вызовы ipset аддона (`create`, `destroy` и проверки `ipset list`), из-за чего гео молча отключалось целиком (`ipset missing, skipping geo`). Теперь аддон при старте находит рабочий `ipset` (проба `ipset version` ловит ту же ошибку линковки) и направляет все вызовы через обёртку: предпочитается **ipset самой прошивки** (`/usr/sbin/ipset` → `/sbin/ipset`) — он согласован со своей libipset, с dnsmasq (который тоже наполняет `awg_dst`) и с модулями ядра `ip_set`; если его нет — берётся Entware `/opt/sbin/ipset` с `LD_LIBRARY_PATH=/opt/lib`. Выбранный бинарь пишется в лог при старте и в `diag`. Это заменяет неполную попытку 1.1.61 (modprobe модулей): на RT-AX58U модули `ip_set` и так были загружены — дело было не в них

### 1.1.62 -- 2026-06-19
- **Защита от самоблокировки роутера при старте VPN (RT-AC68U и др.).** После починки `awg` старт стал доходить до настройки firewall/DNS, и на слабых роутерах это могло отрезать доступ (падал `dnsmasq` → нет DHCP/DNS → «не подключиться ни по Wi-Fi, ни по LAN»). Добавлены четыре предохранителя:
  1. **Дедман-самовосстановление.** Перед рискованной частью старта взводится фоновый «дедман»: если через ~90 c после старта `dnsmasq` мёртв, VPN автоматически откатывается (`stop`) и `dnsmasq` перезапускается — доступ возвращается без физической перезагрузки. Переживает обрыв SSH.
  2. **RAM-aware лимит ipset.** На роутерах с малой памятью (<384 МБ, напр. RT-AC68U 256 МБ) размер набора `awg_dst` ограничивается (98304 вместо 262144), чтобы большие списки (antifilter `ipresolve` ~154K) не съедали всю память и не вешали роутер. Лимит пишется в лог, не молча.
  3. **Безопасный перехват DNS.** `setup_dns_interception` больше не вешает DNAT :53 на роутер, если `dnsmasq` не запущен (иначе у LAN полностью пропал бы DNS).
  4. **Самолечение dnsmasq.** Если после перезапуска `dnsmasq` не поднялся (наш сгенерированный конфиг его сломал), AWG-правила dnsmasq удаляются и `dnsmasq` стартует начисто — лучше урезанный geo, чем мёртвый LAN.
