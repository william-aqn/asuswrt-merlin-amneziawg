# AmneziaWG for Asuswrt-Merlin

![logo](assets/logo.png)


[[русский]](README.md) [[english]](README_EN.md)

💬 Обсуждение и помощь — [Telegram-чат сообщества](https://t.me/asusxray/26094)

VPN-**клиент и сервер** с обходом DPI-блокировок на базе [AmneziaWG](https://github.com/amnezia-vpn/amneziawg-go) с веб-интерфейсом для роутеров ASUS на прошивке [Asuswrt-Merlin](https://www.asuswrt-merlin.net/). Клиент — маршрутизация по устройствам и выборочная маршрутизация через GeoIP/GeoSite. Сервер — входящие подключения к дому из любой точки мира по QR-коду, с той же обфускацией. Обе роли работают одновременно.

Полностью userspace-реализация -- не требует kernel module, работает на любой версии ядра.

> **О проекте:** изначально форк [r0otx/asuswrt-merlin-amneziawg](https://github.com/r0otx/asuswrt-merlin-amneziawg), но с момента ответвления проект очень сильно переработан и развивается самостоятельно. Спасибо r0otx за прекрасную основу.

## Возможности

- **Протокол AmneziaWG** -- WireGuard с обфускацией DPI (Jc, Jmin, Jmax, S1-S4, H1-H4, длинные I1-I5 из AWG 2.x)
- **Userspace-демон** -- на базе [amneziawg-go](https://github.com/amnezia-vpn/amneziawg-go), без kernel module; работает даже на старых ARM32-роутерах с ядром 2.6.x (RT-AC68U) — для них собирается отдельный legacy-демон
- **Веб-интерфейс** -- страница-аддон в стиле ROG (VPN > AmneziaWG), **двуязычный RU/EN** (язык подхватывается из прошивки), плюс **виджет статуса в шапке** всех страниц роутера с быстрым запуском/остановкой туннеля
- **Импорт конфига** -- загрузка `.conf` файла из клиента Amnezia VPN
- **Маршрутизация по устройствам** -- своя политика для каждого устройства: `VPN: весь трафик`, любая гео-политика, `Напрямую`
- **Режим сервера** -- роутер сам принимает входящие AmneziaWG-подключения: доступ с телефона/ноутбука из любой точки к домашней сети и интернету через дом, с полной обфускацией (годится там, где чистый WireGuard режет DPI). Отдельная страница **VPN > AmneziaWG Server**: управление пирами, генерация ключей и **QR прямо в браузере**, per-peer политика маршрутизации (в т.ч. **двойной хоп** — трафик пира уходит в клиентский туннель), умное сосуществование с Xray/XRAYUI. Клиент и сервер работают одновременно (см. раздел [Режим сервера](#режим-сервера--входящие-подключения-v130))
- **Несколько гео-политик** -- независимые гео-политики во вкладках (до 8); в каждой свой набор GeoIP / GeoSite / GeoCustom / Antifilter, устройство выбирает нужную. Одинаковые списки скачиваются в общий пул без дублей
- **Режим политики** -- *включение* (в VPN идёт то, что в списках) или *исключение* (в VPN идёт всё, **кроме** списков)
- **Точечные исключения** -- свои домены/IP/файлы/URL, которые вырезаются из политики (а в режиме исключения, наоборот, добавляются)
- **GeoIP по сервисам** -- IP-диапазоны Telegram, Google, Netflix, Twitter и др. через [Loyalsoldier/geoip](https://github.com/Loyalsoldier/geoip)
- **GeoSite по доменам** -- списки доменов через [v2fly/domain-list-community](https://github.com/v2fly/domain-list-community) + dnsmasq ipset
- **Geo Antifilter** -- готовые списки заблокированных в РФ подсетей/доменов через [antifilter.download](https://antifilter.download/)
- **GeoCustom — свои записи** -- ручное добавление доменов, CIDR-подсетей, своих файлов и URL-источников
- **Анализатор трафика** -- по каждому устройству видно, какие домены/соединения идут через VPN, а какие напрямую; пойманное добавляется в выбранную гео-политику одним кликом
- **Перехват DNS** -- принудительный DNS через dnsmasq, блокировка DoH/DoT для надёжной гео-маршрутизации
- **DNS через туннель** -- опционально весь LAN резолвится через DNS-серверы из конфига внутри туннеля (работают и внутренние DNS VPN-провайдеров вида `100.64.0.1`); DNS-подмена провайдером интернета исключается, при падении туннеля DNS прошивки возвращается автоматически
- **Kill-switch** -- опциональный fail-closed: при внезапной гибели туннеля (крэш демона) трафик устройств с VPN-политикой блокируется, а не уходит открытым в WAN
- **Режим совместимости** -- мирная жизнь с соседними DPI-инструментами (zapret/zapret2, xray/XRAYUI, b4): автообнаружение, работа без перехвата `:53`, отдельная таблица маршрутизации, не пересекающаяся с их правилами
- **Самолечение** -- health-check после старта (пинг/TCP/рукопожатие через туннель) с автооткатом мёртвого туннеля, вотчдог каждые 5 минут (переподнимает упавший туннель, чинит правила после рестарта файрвола), deadman-защита от потери LAN
- **Предупреждения о конфликтах** -- баннеры на страницах: co-resident DPI/прокси (zapret/b4), Xray/XRAYUI (двухрежимный баннер с кнопкой «Остановить Xray» + страж покрытия подсети пиров), VPN-клиент самой прошивки (wgc/VPN Fusion), аппаратное ускорение CTF на старых Broadcom (кнопка «отключить и перезагрузить»), серый/CGNAT WAN-адрес и занятый порт у сервера, доменный Гео без перехвата DNS, чужое match-all правило в dnsmasq
- **Обновления из интерфейса** -- проверка и установка новых версий в один клик (через jsDelivr, когда GitHub заблокирован), установка конкретной версии (откат) и ручная загрузка `.ipk` прямо со страницы; changelog показывается перед обновлением
- **Диагностика** -- кнопка на странице собирает полный отчёт для разбора проблем (платформа, бинарники, маршрутизация, правила, dnsmasq, гео, журналы) с автоматически замазанными ключами
- **MSS clamping** -- автоматическое исправление TCP MSS для туннельного трафика
- **Автообновление** -- ежедневное обновление гео-списков по cron

## Требования

- [Прошивка Asuswrt-Merlin](https://www.asuswrt-merlin.net/download) (`384.15` и новее, `3006.x`)
- Установленный [Entware](https://github.com/Entware/Entware/wiki/Install-on-Asus-stock-firmware) (ставится через [amtm](https://diversion.ch/amtm.html))
- SSH-доступ к роутеру
- Для **режима клиента**: AmneziaWG-сервер, к которому подключаться — конфиг от VPN-провайдера или self-hosted (ставится одной командой: [amneziawg-installer](https://github.com/bivlked/amneziawg-installer))
- Для **режима сервера**: белый (публичный) IP-адрес на WAN — или проброс UDP-порта на вышестоящем роутере (за CGNAT входящие подключения не дойдут; страница предупредит)

## Установка

### Быстрая установка (одна команда)

```shell
curl -sfL https://raw.githubusercontent.com/william-aqn/asuswrt-merlin-amneziawg/main/install-online.sh | sh
```

Скрипт автоматически определит архитектуру роутера, скачает нужный пакет из последнего релиза и установит.

**Если GitHub заблокирован (РФ):** установка через зеркало jsDelivr —

```shell
curl -sfL https://cdn.jsdelivr.net/gh/william-aqn/asuswrt-merlin-amneziawg@main/install-online.sh | sh
```

Скрипт сам скачает `.ipk` через зеркала-прокси. Контрольная сумма SHA256 проверяется, когда доступен GitHub API; при полностью заблокированном GitHub проверка пропускается (jsDelivr раздаёт только файлы репозитория и не может подтвердить хэш релизных пакетов).

### Из .ipk пакета

Скопируйте пакет на роутер и установите:

```shell
scp amneziawg_1.0.0-1_aarch64-3.10.ipk admin@<ip-роутера>:/tmp/
```

```shell
ssh admin@<ip-роутера>
opkg install /tmp/amneziawg_1.0.0-1_aarch64-3.10.ipk
```

### Ручная установка

Проект полностью **userspace**: демон `amneziawg-go` + утилита `awg`, без кастомного модуля ядра (нужен только штатный `tun`). Для чистой установки проще всего собрать `.ipk` (`./build-ipk.sh`) и поставить его через `opkg` (см. раздел «Из .ipk пакета») — пакет сам разложит бинарники в `/opt/amneziawg`, аддон в `/jffs/addons/amneziawg`, создаст init-скрипт `S99amneziawg` и зарегистрирует страницу.

Для быстрого обновления **только аддона** (бэкенд-скрипт и веб-страница) без пересборки бинарников — скопируйте файлы прямо в `/jffs/addons/amneziawg/` и перезапустите:

```shell
scp addon/amneziawg.sh addon/amneziawg_page.asp addon/amneziawg_widget.js \
    addon/amneziawg_server.sh addon/amneziawg_server_page.asp addon/awg_qr.js \
    admin@<ip-роутера>:/jffs/addons/amneziawg/
ssh admin@<ip-роутера>
/jffs/addons/amneziawg/amneziawg.sh install_page    # перерегистрировать страницы в меню роутера
/jffs/addons/amneziawg/amneziawg.sh restart          # перезапустить клиентский туннель с новым кодом
/jffs/addons/amneziawg/amneziawg_server.sh restart   # (если запущен сервер) перезапустить и его
```

> SCP-клиенты с графическим интерфейсом: для Windows — [WinSCP](https://winscp.net/eng/download.php), для macOS — [MacSCP](https://www.macscp.co/).

### После установки

1. Выйдите и войдите заново в веб-интерфейс роутера
2. **Клиент** (выход в интернет через внешний VPN): **VPN > AmneziaWG** → **Import Config** → загрузите `.conf` из клиента Amnezia VPN → **Apply**
3. **Сервер** (входящие подключения к дому): **VPN > AmneziaWG Server** → «Сгенерировать» ключи → добавьте пира → **Применить** → **Запустить сервер** → отсканируйте QR приложением AmneziaWG

Роли независимы: можно использовать любую по отдельности или обе сразу.

## Использование

### Быстрый старт (клиент)

1. Экспортируйте конфиг из клиента Amnezia VPN (файл `.conf`)
2. В интерфейсе роутера: **VPN > AmneziaWG > Import Config** -- загрузите файл
3. Нажмите **Apply** -- туннель запустится автоматически
4. Добавьте устройства в разделе **Device Rules** с нужной политикой

Входящие подключения (доступ к дому со своих устройств) настраиваются отдельно — см. [Режим сервера](#режим-сервера--входящие-подключения-v130).

### Политики маршрутизации

В «Правилах устройств» у каждого устройства выбирается политика:

| Политика | Описание |
|----------|----------|
| **VPN: весь трафик** | Весь трафик устройства через VPN |
| **VPN: \<имя гео-политики\>** | Маршрутизация по выбранной гео-политике (см. ниже). Таких пунктов столько, сколько создано гео-политик |
| **Напрямую** | Устройство обходит VPN |

### Гео-политики (вкладки)

Гео-политик может быть несколько (до 8) — они задаются **вкладками** (кнопка «+ Добавить гео политику»). В каждой вкладке свой независимый набор источников: **GeoIP**, **GeoSite**, **GeoCustom** (свои домены/IP/файлы/URL) и **Geo Antifilter**. Какое устройство по какой политике ходит — выбирается в «Правилах устройств».

Все политики идут в **один** туннель — различается только то, *какие* адреса в него попадают (у каждой политики свой `ipset`). Одинаковые списки в разных политиках скачиваются и хранятся **один раз** (общий пул); в туннель каждой политики попадает только выбранное именно в ней. На слабых роутерах общий бюджет `ipset` делится между политиками, чтобы не исчерпать память.

**Режим политики** — переключатель в начале вкладки:

| Режим | Поведение |
|-------|-----------|
| **В VPN по спискам** (включение) | В туннель идёт только совпавшее со списками; остальное — напрямую |
| **Напрямую по спискам** (исключение) | Наоборот: в туннель идёт весь трафик устройства, **кроме** того, что в списках |

> Режим исключения тянет почти весь трафик в туннель (как «весь трафик», но с вырезами) — учитывайте при сосуществовании с zapret/Xray/b4.

**Точечные исключения** — блок в конце вкладки (свои домены/IP/подсети/файлы/URL). Записи в нём ведут себя как исключения с учётом режима: в режиме включения идут **напрямую** (вырезаются из VPN), в режиме исключения — **в VPN**. Удобно, когда большой список (например категория GeoSite) тянет лишнее.

### GeoIP Service Lists

Введите имена сервисов через запятую для маршрутизации их IP-диапазонов через VPN:

```
telegram,google,facebook,twitter,netflix,cloudflare
```

Работают по IP -- не зависят от DNS, идеальны для Telegram и других приложений, которые подключаются напрямую по IP.

### GeoSite Service Lists

Введите имена сервисов для маршрутизации по доменам через dnsmasq:

```
youtube,google,discord,netflix,spotify,instagram
```

Требует чтобы устройства использовали роутер как DNS-сервер. Для iPhone: **Настройки > Wi-Fi > (i) > DNS > Вручную > только IP роутера**.

### Geo Antifilter

Готовые списки заблокированных в РФ ресурсов с [antifilter.download](https://antifilter.download/). Отмечайте нужные галочками — IP-списки грузятся в ipset этой гео-политики (рядом с GeoIP) и маршрутизируются через VPN:

- **allyouneed** (~15K) -- все нужные подсети (= `ipsum` + `subnet`); рекомендуемый выбор
- **community** (~900) -- подсети сообщества
- **ipsum** / **subnet** / **ip** (~48K) -- альтернативные срезы, сильно пересекаются с `allyouneed`
- **ipresolve** (~154K) -- IP из DNS-резолва; очень большой список
- **community домены** (~485) -- небольшой список доменов → dnsmasq

Полный `domains.lst` (1.4M доменов / 27 МБ) намеренно не подключается — он слишком велик для dnsmasq на роутере. Обновляются той же кнопкой **Update Now** и автообновлением, что и остальные гео-списки.

### GeoCustom — свои записи

- **Свои домены** -- домены через запятую/с новой строки (например `example.com,service.org`)
- **Свои IP / подсети** -- IP/CIDR через запятую (например `8.8.8.8,1.1.1.0/24`)
- **Свои файлы** -- именованные списки, которые можно вставить/отредактировать прямо в интерфейсе или загрузить из файла (домен → DNS, IP/подсеть → ipset; строки с `#` — комментарии)
- **URL-источники** -- ссылки на скачиваемые списки того же формата

Те же поля есть в блоке **«Точечные исключения»** — но там они работают как исключения (см. [Гео-политики](#гео-политики-вкладки)).

### Режим сервера — входящие подключения (v1.3.0)

Роутер может работать не только клиентом, но и **AmneziaWG-сервером**: устройство (телефон, ноутбук, второй роутер) из любой точки подключается к дому — получает доступ к домашней сети (NAS, камеры, RDP) и, по желанию, выходит в интернет через ваш дом. Всё с обфускацией AmneziaWG, поэтому проходит там, где обычный WireGuard блокируется DPI. Целевые клиенты — официальные приложения **AmneziaWG 2.x** (Android/iOS/Windows/macOS).

Отдельная страница **VPN > AmneziaWG Server** (клиентская часть не затрагивается, обе роли можно держать включёнными одновременно):

- **Настройка** — «Сгенерировать» пару ключей сервера, порт (по умолчанию `51821`, чтобы не конфликтовать со встроенным WG-сервером прошивки), подсеть туннеля (по умолчанию `10.9.0.0/24`), MTU. Параметры обфускации (`Jc/Jmin/Jmax/S1/S2/H1-H4`, кнопка «Сгенерировать случайные») + продвинутые `I1-I5` для AWG 2.x.
- **Пиры** — таблица устройств: имя, авто-выданный IP, вкл/выкл, **политика маршрутизации** (`Напрямую` / `VPN: весь трафик` / любая гео-политика), **что в туннель** (весь трафик или только домашняя сеть). Ключи пары и PSK генерируются **прямо в браузере** (Curve25519) — приватные ключи пиров не уходят на бэкенд.
- **Выдача конфигов** — на каждого пира: **QR-код** (свой генератор, без внешних CDN; сканируется приложением AmneziaWG), скачивание `.conf`, копирование.
- **Per-peer политика (двойной хоп)** — пиру можно назначить ту же политику, что и LAN-устройству: с `VPN: весь трафик` трафик пира уходит в **клиентский** туннель (телефон → дом → внешний VPN). Работает поверх того же движка политик. Правила живут вместе с клиентским фаерволом: пока клиентский туннель выключен, трафик пира идёт напрямую в WAN (fail-open, о чём предупреждает жёлтый баннер).
- **DNS для пиров** — по умолчанию dnsmasq роутера (работают имена домашних устройств и доменный Гео), либо свои серверы.
- **NAT в LAN** (по умолчанию вкл) — пиры видны LAN-машинам как сам роутер, что лечит молчание Windows-фаервола на «чужую подсеть».
- **Предупреждения** — баннеры при сером/CGNAT WAN-адресе (сервер не примет входящие — нужен белый IP или проброс порта), при конфликте порта со встроенным WG-сервером прошивки и о сосуществовании с Xray/XRAYUI (см. ниже).

**Сосуществование с Xray / XRAYUI** (проверено на живом роутере с обоими):

- **XRAYUI в режиме «весь трафик» + пиры «Напрямую» — штатная рабочая схема.** XRAYUI сам подхватывает подсеть пиров (ставит для неё те же TPROXY-правила, что и для LAN), поэтому трафик пиров автоматически идёт **через Xray** (обход DPI), непроксируемые адреса — напрямую в WAN. Страница показывает жёлтый информационный баннер.
- **Двойной хоп при работающем Xray недоступен**: Xray перехватывает трафик раньше правил AmneziaWG (ip-rule 19 против 99) и глушит клиентский туннель. Если у пиров стоят VPN-политики при активном Xray — красный баннер с двумя рабочими вариантами: оставить Xray и перевести пиров на «Напрямую», либо остановить Xray (кнопка в баннере), поставить пиру «VPN: весь трафик» и запустить клиентский туннель.
- **Страж покрытия**: если Xray запущен, но его правила перехвата не включают подсеть пиров (обычно XRAYUI стартовал раньше сервера) — трафик пиров идёт мимо Xray напрямую в WAN, без обхода DPI. Аддон это замечает (лог + жёлтый баннер) и подсказывает решение: перезапустить XRAYUI (при старте он подхватывает существующие интерфейсы) или добавить подсеть в его настройки.

CLI: `/opt/etc/init.d/S99amneziawg server {start|stop|status|restart|diag}`. Автозапуск после перезагрузки — **отдельный, по умолчанию выключенный** тумблер (сервер, открывающий порт в WAN, не поднимается сам).

## Сборка из исходников

### Необходимо

- Docker Desktop
- Go (свежий stable; для пакета `armv7-2.6` дополнительно нужен тулчейн Go 1.23 — см. ниже)
- GNU tar (`brew install gnu-tar` на macOS)

### Сборка amneziawg-go (userspace-демон)

> **Демон собирается из нашего форка [`william-aqn/amneziawg-go`](https://github.com/william-aqn/amneziawg-go), а не из upstream.** Причина — два фикса, критичных для роутеров, которые ещё не приняты в upstream (пока они висят в PR, сборка идёт из форка):
> - **[PR #152](https://github.com/amnezia-vpn/amneziawg-go/pull/152)** — ограниченный буферный пул (`PreallocatedBuffersPerPool`): лекарство от `runtime: out of memory` под нагрузкой на роутерах с малой памятью;
> - **[PR #153](https://github.com/amnezia-vpn/amneziawg-go/pull/153)** — fallback `sendmmsg`/`recvmmsg` → пакетный `sendmsg`/`recvmsg` при `ENOSYS`: без него на ядре Linux < 3.0 (RT-AC68U / 2.6.36) демон не может отправить ни одного пакета и туннель не передаёт трафик.
>
> Ветка форка **`router-build`** = тег `v0.2.19` (те же AmneziaWG 1.5/2.0 параметры) + оба патча отдельными коммитами. **Когда оба PR примут в upstream**, сборка вернётся на `amnezia-vpn/amneziawg-go` — правятся две строки (`AWG_GO_REPO`/`AWG_GO_REF`) в `.github/workflows/release.yml`.

```shell
git clone --depth 1 --branch router-build https://github.com/william-aqn/amneziawg-go.git
cd amneziawg-go

# ARM64 (aarch64-3.10) — GT-AX11000, RT-AX86U, RT-AX88U
CGO_ENABLED=0 GOOS=linux GOARCH=arm64 go build -ldflags="-s -w" -o ../output/amneziawg-go

# ARM32 (armv7-3.2) — RT-AX56U, RT-AX58U, новые HND-роутеры
CGO_ENABLED=0 GOOS=linux GOARCH=arm GOARM=7 go build -ldflags="-s -w" -o ../output/amneziawg-go-arm
```

**ARM32 `armv7-2.6` (RT-AC68U, RT-AC66U и другие роутеры на ядре 2.6.36) собирается иначе.** Go ≥ 1.24 поддерживает только ядро Linux ≥ 3.2 — обычная сборка на таком роутере молча падает сразу после запуска. Поэтому демон для этого пакета собирается тулчейном **Go 1.23** (поддерживает ядра с 2.6.32) с откатом зависимостей до 1.23-совместимых:

```shell
go get golang.org/x/sys@v0.35.0 golang.org/x/net@v0.43.0 golang.org/x/crypto@v0.41.0
go mod edit -go=1.23.0 -toolchain=none
GOTOOLCHAIN=go1.23.12 CGO_ENABLED=0 GOOS=linux GOARCH=arm GOARM=5 \
  go build -ldflags="-s -w" -o ../output/amneziawg-go-arm5
```

Канонические команды (включая патч версии, чтобы `--version` показывал `v0.2.19-legacy26-pool1024-smfix`, и жёсткие проверки наличия обоих патчей) — в `.github/workflows/release.yml`, шаг «Build amneziawg-go-arm5 (legacy Go 1.23…)».

### Сборка awg CLI (статический musl)

Утилита `awg` линкуется статически с musl (чтобы работать на старой glibc роутера). Канонические команды для всех вариантов — в `.github/workflows/release.yml` (шаги «Build awg…»); CI собирает `awg`/`awg-arm`/`awg-arm5` и публикует `.ipk`. Локально для ARM64:

```shell
docker run --rm --platform linux/arm64 -v "$PWD/output:/out" alpine:3.19 sh -c \
```

> ARM32: `armv7` — то же с `--platform linux/arm/v7`; старый `armv5` (RT-AC68U) **обязательно** soft-float (`dockcross/linux-armv5-musl`), иначе SIGILL на ядрах без VFP. Точные команды — в `release.yml`.

## Архитектура

**Клиент** (исходящий туннель):

```
Интернет <-- awg0 (туннель) <-- iptables mangle цепочка AWG <-- br0 (устройства LAN)
                                          |
                            ipset каждой гео-политики: awg_dst, awg_dst2, …
                            (GeoIP/Antifilter CIDR + DNS-резолвы; у политики
                             с исключениями ещё свой awg_dst<id>_x)
                                          |
                                  fwmark 0x100 -> таблица маршрутов 300 -> awg0
```

**Сервер** (входящие подключения; свой интерфейс и демон, клиентская часть не затрагивается):

```
Телефон/ноутбук (пир) --> WAN UDP:51821 --> awgs0 (сервер) --> br0 (домашняя сеть)
                                               |                └ NAT в LAN (опция)
                                               ├--> напрямую в WAN (политика «Напрямую»)
                                               └--> в клиентский туннель awg0 (двойной хоп,
                                                    политика «VPN» — тот же движок политик)
```

| Компонент | Назначение |
|-----------|-----------|
| **amneziawg-go** | Userspace WireGuard-демон с расширениями AmneziaWG (клиентский туннель `awg0`) |
| **awgs-go** | Тот же демон под отдельным именем процесса — серверный туннель `awgs0` (жёсткая ссылка, чтобы жизненные циклы ролей не пересекались) |
| **awg** | CLI-утилита для управления туннелями |
| **amneziawg.sh** | Backend клиента: жизненный цикл, firewall, маршрутизация, гео-списки, перехват DNS; общая библиотека хелперов для обеих ролей |
| **amneziawg_server.sh** | Backend сервера: пиры, firewall входящих, DNS для пиров, per-peer политика (использует хелперы `amneziawg.sh`) |
| **amneziawg_page.asp** | Веб-интерфейс клиента (**VPN > AmneziaWG**) |
| **amneziawg_server_page.asp** | Веб-интерфейс сервера (**VPN > AmneziaWG Server**) |
| **awg_qr.js** | Генерация QR-кодов и ключей Curve25519 прямо в браузере (без внешних CDN; приватные ключи пиров не покидают страницу) |
| **amneziawg_widget.js** | Глобальный виджет в шапке роутера: индикатор статуса ● AWG на всех страницах прошивки + мини-панель включения/выключения туннеля (отдаётся как `/www/user/awg_widget.js`, подгружается через `menuTree.js`) |

## FAQ

**В: Telegram не работает через VPN?**

О: Добавьте `telegram` в GeoIP Service Lists. Telegram подключается по IP напрямую -- доменных списков недостаточно.

**В: Сайты не открываются на iPhone с гео-политикой?**

О: iPhone использует зашифрованный DNS (DoH), который обходит dnsmasq роутера. Настройте DNS вручную: Настройки > Wi-Fi > (i) > DNS > Вручную > только IP роутера.

**В: Туннель работает для ping, но сайты не открываются?**

О: Перезапустите туннель с паузой: `/jffs/addons/amneziawg/amneziawg.sh stop; sleep 5; /jffs/addons/amneziawg/amneziawg.sh start`

**В: Как добавить свой сервис по IP?**

О: Добавьте CIDR-диапазоны в поле «Свои IP / подсети» (GeoCustom), например `149.154.160.0/20,91.108.4.0/22` для Telegram.

**В: Поддерживается ли ARM32 (RT-AC68U)?**

О: Да, есть отдельный .ipk для ARM32 (`armv7-2.6`). Начиная с **1.2.32** демон в этом пакете собирается специальным legacy-тулчейном (Go 1.23) — обычные сборки Go ≥ 1.24 не поддерживают ядро 2.6.36 этих роутеров и молча падали с `ERROR: amneziawg-go failed to create interface`. Проверить, что стоит нужная сборка: `/opt/amneziawg/amneziawg-go --version` должен показывать `v0.2.19-legacy26-pool1024-smfix (…)` (демон собран из [форка](https://github.com/william-aqn/amneziawg-go) с двумя фиксами — см. раздел «Сборка amneziawg-go»; суффикс `-smfix` = фикс `sendmmsg`, без которого туннель на 2.6.36 не передаёт трафик).

**В: Туннель сам останавливается через минуту-две после запуска (или «2 минуты работает → дроп → переподключение»)?**

О: Обновитесь до **1.2.32+**. Раньше health-check и вотчдог проверяли туннель пингом (и TCP) до узлов из настройки `awg_watchdog_hosts` — и если указанные узлы не отвечают ни на ICMP, ни на TCP:443 (Cloudflare WARP, внутренние шлюзы провайдера вида `100.64.0.1`), полностью рабочий туннель ложно признавался мёртвым и откатывался. Теперь доказательством жизни служит само **рукопожатие WireGuard** (наши же пробные пакеты заставляют его обновляться), так что «молчаливые» узлы больше не приводят к откатам. Поле «Хосты проверки» при этом лучше оставить пустым (по умолчанию `8.8.8.8 1.1.1.1`) или указать реально пингуемые адреса — с ними проверка проходит быстрее.

**В: Как временно отключить туннель, чтобы он не поднимался сам после перезагрузки (не удаляя аддон)?**

О: С **1.2.52** — штатно: остановите туннель кнопкой «Остановить» и снимите галочку **«Автозапуск»** в настройках (Применить). После перезагрузки туннель останется остановленным, весь рабочий конфиг цел; когда снова понадобится — «Запустить» и, при желании, верните галочку. Удобно на время отладки соседних программ (zapret2, Xray и т.п.).

**В: Можно ли использовать вместе с zapret2 или Xray (XRAYUI)?**

О: Да, но с оговорками. Аддон автоматически определяет рядом работающие DPI-обходы/прокси (zapret2/bol-van, Xray/XRAYUI, v2ray, sing-box, **b4**, а также правила NFQUEUE/TPROXY в iptables или nftables) и в этом случае **не включает перехват DNS** (DNAT на :53), чтобы не конфликтовать с ними -- иначе сеть может остаться без интернета. Этим управляет **«Режим совместимости»** (бывшая галочка «Не перехватывать DNS»): при чистой установке он включён по умолчанию, чтобы даже неизвестная утилита не оставила сеть без интернета; у обновлённых установок поведение прежнее, пока вы сами его не включите.

Важно: это снимает только конфликт по DNS. При политике по умолчанию **«VPN -- весь трафик»** маршрутизация всё равно заберёт трафик у соседнего прокси, поэтому для совместимости выбирайте политику **«Напрямую»** или **«VPN -- только Geo»**, а не «весь трафик». Geo-маршрутизация по IP при этом продолжает работать.

**Обратный случай — XRAYUI забирает трафик AmneziaWG.** Если XRAYUI работает в режиме прозрачного проксирования «перенаправить весь трафик» (TPROXY), он перехватывает и собственный egress роутера — включая рукопожатие AmneziaWG, — и тогда туннель поднимается, но трафик не идёт (health-check откатывает его примерно через 60 с). Аддон это определяет и показывает на странице **красный баннер**. Решение: в XRAYUI отключить режим «redirect all» / прозрачную маршрутизацию, либо исключить из перехвата endpoint и интерфейс `awg0`, либо держать включённым только один VPN за раз.

**В: Что нужно, чтобы работал режим сервера (подключение к дому)?**

О: Белый (публичный) IP на WAN — или проброс UDP-порта аддона (по умолчанию 51821) на вышестоящем роутере. За CGNAT/серым адресом входящие подключения физически не дойдут — страница сервера предупредит красным баннером. Адрес для конфигов пиров подставляется автоматически (DDNS прошивки, иначе текущий WAN IP); клиентом на устройстве выступает официальное приложение **AmneziaWG 2.x** (сканирует QR со страницы). Автозапуск сервера после перезагрузки — отдельная галочка, по умолчанию выключена.

**В: Подключился к серверу роутера — рукопожатие есть, а у пира с политикой «VPN» нет интернета?**

О: Проверьте баннеры на странице сервера. Частые причины: клиентский туннель не запущен (политика «VPN» у пира — это двойной хоп через него; пока он лежит, трафик идёт напрямую — fail-open), либо работает Xray/XRAYUI в режиме «весь трафик» — он несовместим с двойным хопом (перехватывает трафик раньше правил аддона и глушит клиентский туннель). С Xray держите пиров на «Напрямую» — их трафик и так пойдёт через Xray (страница подскажет, если Xray не подхватил подсеть пиров — лечится перезапуском XRAYUI).

**В: Совместим ли аддон с QoS (Adaptive QoS / Cake / традиционный)?**

О: Да. Проверено вживую на GT-AX6000 с Adaptive QoS (Trend Micro): для QoS весь туннель — один зашифрованный UDP-поток от роутера, он классифицируется и шейпится целиком (метка bwdpi появляется прямо на conntrack-записи туннеля), а скорость через туннель при включении QoS не изменилась (22.7 → 22.2 МБ/с — в пределах погрешности). Включение/выключение QoS перезапускает файрвол — правила аддона переустанавливаются автоматически (хук firewall-start), туннель не падает.

Честные ограничения:
- **per-app и per-device приоритеты QoS не действуют на трафик внутри туннеля** — QoS не видит, что там YouTube с телевизора; это свойство любого VPN на роутере, а не аддона. Приоритизация/лимит туннеля как целого — работает; direct-устройства классифицируются как обычно;
- на слабых роутерах учитывайте суммарную нагрузку CPU: DPI-движок QoS + userspace-шифрование amneziawg-go;
- маловероятный, но возможный конфликт по пакетным меткам актуален в первую очередь для **традиционного QoS** (Adaptive на HND-платформах метит на уровне ядра/conntrack и в iptables перед нашей цепочкой не встаёт). Симптом однозначный: после включения QoS гео-устройства перестают ходить через туннель — снимите диагностику (там видно цепочку AWG и счётчики) и напишите в чат.

**В: `ipset` пишет `Warning: Kernel support protocol versions 6-6 while userspace supports protocol versions 6-7`?**

О: Это безвредное предупреждение, а не ошибка. Утилита `ipset` (и прошивочная `/usr/sbin/ipset`, и Entware `/opt/sbin/ipset`) собрана из более свежих исходников и умеет протокол версий 6 и 7, а модуль ipset в старом ядре роутера (4.1.51) реализует только версию 6. Диапазоны пересекаются на версии 6, поэтому утилита автоматически работает по ней -- всё функционирует штатно. Делать ничего не нужно; geo-роутинг через ipset (`awg_dst` и geo-ipset'ы) полностью работает на версии 6.
